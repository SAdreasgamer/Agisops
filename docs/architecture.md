# AEGISOPS Phase 1: Foundation & Infrastructure Architecture

This document describes the architectural layout, core components, data flows, and schemas implemented in Phase 1.

---

## 1. System Overview

AEGISOPS is a multi-tenant, AI-driven incident investigation and remediation orchestrator. Phase 1 establishes the baseline infrastructure including:
- **Webhook Ingestion Layer** (Spring Boot)
- **Message Broker & Event Streaming** (Apache Kafka)
- **Workflow Orchestration & DB Persistence** (Spring Boot + PostgreSQL + Flyway)
- **Security & Route Management** (Spring Cloud Gateway + Spring Security + OAuth/RBAC)
- **Cross-Language gRPC Pipeline** (Protobuf contract sharing between Spring Boot and Python AI services)

## 2. Ingestion Layer

The Ingestion Service accepts alerts and events from external systems (GitHub, PagerDuty, AlertManager) and forwards them to a raw events Kafka stream after validation and deduplication.

### Endpoint Structure
- `POST /api/v1/ingest/github` — Processes GitHub repository webhooks.
- `POST /api/v1/ingest/pagerduty` — Processes PagerDuty incident alerts.
- `POST /api/v1/ingest/alertmanager` — Processes Prometheus AlertManager alerts.

### HMAC Signature Verification
For GitHub webhooks, security is enforced via HMAC SHA-256 signatures passed in the `X-Hub-Signature-256` header:
- Signature is generated by calculating a HMAC-SHA256 signature using the raw request body payload and a pre-shared webhook secret key.
- Unsigned or invalid signature requests are rejected with a `401 Unauthorized` status.

### Redis-Based Deduplication
To ensure idempotency and prevent duplicate event processing:
- A deduplication key is calculated using the event source and its unique identifier (e.g. `dedup:github:pr-12345`).
- Redis `SETNX` (set if absent) is used with a configurable TTL (e.g. 5 minutes).
- If the key exists, the request returns a `200 OK` with status `DUPLICATE` without publishing to Kafka. Otherwise, it is labeled `ACCEPTED`.

## 3. Message Queue & Serialization

All parsed events are published to a central Kafka topic: `aegisops.events.raw`.

### Event Schema (`AegisEvent`)
Every event is wrapped in a structured metadata envelope:
- `eventId`: UUID identifying the ingestion event.
- `eventType`: Type of alert (e.g. `pull_request.opened`, `incident.triggered`).
- `timestamp`: Date and time of ingestion.
- `source`: The originating system (`GITHUB`, `PAGERDUTY`, `ALERTMANAGER`).
- `tenantId`: Identifies the organization tenant scope.
- `payload`: Map of raw JSON key-values from the source payload.
- `metadata`: Harmonized fields extracted from the payload (e.g., `title`, `description`, `incident_id`).

### Kafka Configuration
- **Producers**: Configured as idempotent with a validation strategy, using a `JsonSerializer` for the event payload.
- **Consumers**: Configured with explicit consumer factory and listener container factories to auto-register deserializers and trust class packages matching model packages.

## 4. Persistence Layer

The database schema is managed using **Flyway Migrations** in the `common` module.

### Core Tables
1. **`tenants`**: Represents customer partition scope.
2. **`events`**: Log of all successfully ingested events.
3. **`incidents`**: Tracked system incidents mapped from ingested raw alerts.
4. **`workflows`**: Workflow executions started by the orchestrator.
5. **`workflow_steps`**: Tracked executions of subtasks (agents or manual steps).
6. **`audit_logs`**: Append-only log tracking actor behaviors for compliance.

## 5. Security & Gateway Routing

The API Gateway manages authentication and route proxying.

### Routing Table
- `/api/v1/auth/**` -> Forwarded to `auth-service`.
- `/api/v1/workflows/**` -> Forwarded to `workflow-orchestrator`.
- `/api/v1/ingest/**` -> Route direct access to `ingestion-service`.

### Security Filters
- **JWT Verification Filter**: Inspects the `Authorization` header, parses the signature, and populates the Security Context.
- **RBAC**: Enforces role access at endpoints:
  - `SRE`: Full access to trigger workflows and query incidents.
  - `Developer`: Read-only access to workflow statuses.
  - `Administrator`: Tenant-level setting management.

## 6. Cross-Language gRPC Setup

Service communication uses Protobuf-based contracts to coordinate between Spring Boot and Python AI Services.

### Protocol Buffers (`proto/aegisops/`)
- `common.proto`: Shared types, error codes, and state definitions.
- `retrieval.proto`: Context retrieval requests from vector database (`RetrievalService`).
- `agent.proto`: Orchestrator-to-agent task submission contracts.

### Code Generation Workflow
1. **Java**: Generated during compiling using `protobuf-maven-plugin`. Classes populate `target/generated-sources/`.
2. **Python**: Compiled using `compile_protos.sh` (wraps `grpcio-tools`) and outputs Python stubs in `python-services/gen/aegisops/`.
